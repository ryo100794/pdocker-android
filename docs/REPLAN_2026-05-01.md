compatibility, project-library, PRoot, networking, resident-service, and GPU
steering work.

- Android APK build passes with the current submodule and staged pdockerd.
- Offline compatibility audit passes with 53 PASS, 0 FAIL, 0 SKIP.
- UI action wiring audit passes, including daemon readiness wait, legacy builder
  env, detached compose up, terminal palette, editor tools, and image deep-link.

## Priority Plan

### P0: Make current UI actions observably reliable

1. Add a native job model for Docker actions: **first pass landed**
   - job id, command, project/container target, status, start/end time, exit code
   - progress lines from `docker build`
   - tail log buffer
   - stop, retry, and log tabs landed; pending: richer progress parsing and open terminal after process death
2. Replace `compose up` menu result with a job card in the upper pane: **first pass landed**
3. Add APK/device smoke script for:
   - start pdockerd
   - build tiny Dockerfile
   - compose up/down tiny project
   - inspect UI-visible state/log files

Acceptance:

- User can start build/compose from UI, leave the terminal tab, and still see
  progress and final status in widgets.
- Tiny build and tiny compose scenario are recorded as repeatable tests.

### P1: Close the biggest Docker compatibility holes users hit first

1. Implement Engine-side PTY attach for `docker run -t` and `docker exec -it`.
2. Make unsupported port publishing explicit: **first pass landed**
   - warning in create/start response metadata
   - UI badge on containers
   - docs/tests for host-network-only behavior
3. Improve Dockerfile build robustness:
   - split build execution from HTTP handler
   - preserve `.dockerignore` basics
   - add time/progress checkpoints around clone/materialize/snapshot

Acceptance:

- `docker exec -it <cid> sh` from bundled CLI works through pdockerd, not only
  via app-created PTY sessions.
- `docker run -p` no longer looks silently supported.

### P2: File management as a real workflow

1. Add project/file tree panel for Compose, Dockerfile, workspace, templates:
   **recent editable project/import files surfaced; full tree still pending**
2. Add container rootfs browser and log browser: **read-only container browser landed**
3. Add safe edit/copy operations:
   - read-only image rootfs by default
   - editable project files
   - copy from image/container to project: **landed**
   - explicit write/copy into container upperdir when supported: **landed**
4. Add search/replace and multi-file tabs to the editor: **search/replace landed; multi-file tabs already available in lower pane**

Acceptance:

- User can inspect image/container files, copy them into a project, edit them,
  and rebuild without leaving the app UI.

### P3: GPU path from contract to measurement

1. Add Android NDK `android-gpu-bench` module:
   - `cpu_scalar`, `cpu_neon`, `vulkan_spirv`
   - `vector_add`, `saxpy`, `matmul_fp32`
   - JSONL/CSV result writer
   - device info collector
2. Add Vulkan probe inside pdocker GPU containers:
   - `vulkaninfo`/loader diagnostics
   - visible UI status
   - fallback explanation
3. Start cuVK runtime skeleton:
   - `cuvkInit`, `cuvkMalloc`, `cuvkMemcpy`, `cuvkLaunchKernel`
   - pre-generated GLSL/SPIR-V path before source transpiler

Acceptance:

- The app can answer: "On this Android device, what size makes Vulkan faster
  than CPU NEON?"
- `cuda-compat` has a real minimal API path, not only environment variables.

### P4: PRoot retirement path

1. Introduce RuntimeBackend boundary in pdockerd.
2. Keep PRoot as `ProotBackend`.
3. Add no-PRoot packaging tests that still pass pull/list/edit/browse flows.
4. Prototype replacement executor for non-interactive start/wait/logs.
5. Move filesystem lower/upper semantics out of patched PRoot.

Acceptance:

- Default APK can be built without PRoot/talloc and still expose non-runtime
  workflows.
- A replacement backend can run a minimal container command before PRoot is
  removed from the default runtime.
